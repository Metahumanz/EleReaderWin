# 本地文本分章逻辑文档

本文档摘录了项目（Legado/阅读）中关于本地 `.txt` 及类似格式文件的分章逻辑、核心算法及默认规则。

## 1. 编码检测

为了正确读取本地文本，App 首先会检测文件的字符编码：
- **核心逻辑**：读取文件前 512KB 数据，使用 `EncodingDetect.getEncode(buffer)` 进行识别。
- **支持范围**：支持 UTF-8 (含 BOM)、GBK、Big5 等常见编码。

## 2. 目录规则匹配 (TOC Rule)

项目内置了一套基于正则表达式的目录识别系统。

### 2.1 规则选择流程
1. **采样**：读取文件起始部分的 512KB 内容。
2. **测试**：遍历所有已启用的 `TxtTocRule`（预设规则见 `txtTocRule.json`）。
3. **计分**：统计每个正则在此 512KB 内容中匹配出的“章节”数量。
4. **决策**：匹配数量最多的规则（且至少匹配到 1 个）将被选为该书的分章规则。

### 2.2 默认正则示例
| 规则名称 | 正则表达式 | 匹配示例 |
| :--- | :--- | :--- |
| **标准目录** | `^[ 　\t]{0,4}(?:序章\|楔子\|第\s{0,4}[...]+?\s{0,4}(?:章\|节\|卷)).{0,30}$` | `第一章 开端` |
| **数字分隔符** | `^[ 　\t]{0,4}\d{1,5}[:：,.， 、_—\-].{1,30}$` | `1、起始章节` |
| **特殊符号** | `(?<=[\s　])[【〔〖「『〈［\[](?:第\|Chapter)...[章节].{0,20}$` | `【第一章】` |

## 3. 分章算法实现

核心处理类为 `TextFile.kt`，采用流式读取和二次拆分策略。

### 3.1 缓冲分块处理
- 使用 **512KB** 的 `bufferSize` 进行分块读取。
- 在块与块之间，通过查找换行符 (`0x0a`) 来保证分块不会剪断一行。

### 3.2 逻辑分支
1. **有规则匹配**：
    - 按正则分割内容。
    - **前言处理**：第一个匹配项之前的文本会被标记为“前言”。
    - **卷标识别**：如果匹配行之后内容为空，则标记为 `isVolume`。
2. **模糊分章 (无规则)**：
    - 如果所有正则均不匹配，按固定长度（默认 **10KB**）强制分章。
    - 同样会寻找换行符作为分章点，以防截断段落。

### 3.3 超大章节二次拆分 (Sub-splitting)
- **触发阈值**：单个章节长度超过 **100KB** (有规则) 或 **10KB** (无规则)。
- **逻辑**：递归调用 `analyze` 函数，将过长章节拆分为 `章节名(1)`, `章节名(2)` 等子章节，确保阅读器性能。

## 4. 核心代码参考

- **分块逻辑**：`TextFile.getChapterList()`
- **正则匹配逻辑**：`TextFile.getTocRule(content: String)`
- **内容提取**：`TextFile.getContent(chapter: BookChapter)`
  - 通过 `start` 和 `end` 字节偏移量直接定位文件流，避免 OOM。
